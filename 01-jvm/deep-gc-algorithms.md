# GC 演算法比較（G1、ZGC）

## 前言

[GC 基礎篇](gc-basics-and-generations.md)收在一句話：「現代收集器的演化方向是把大停頓拆小、把工作挪到與應用並行的階段」。這篇把話說完：**具體怎麼拆、怎麼挪？**

先亮同場實測（同一份負載、1GB heap、只換收集器）：

| 收集器 | 總耗時 | 停頓次數 | **最長停頓** |
|---|---|---|---|
| Serial | 368 ms | 28 | 37.5 ms |
| Parallel | 232 ms | 24 | 16.3 ms |
| G1 | 325 ms | 62 | **9.2 ms** |
| ZGC | 156 ms | 21 | **0.005 ms** |

從 37.5ms 到 **5 微秒**——四個收集器就是四種「吞吐 vs 延遲」的答案。這篇講每一種的核心機關，以及怎麼替你的服務選。

## 技術背景

### 評估 GC 的三角習題

**吞吐**（应用做正事的時間占比）、**延遲**（單次停頓多長）、**footprint**（GC 自身吃多少記憶體與 CPU）——三者不可兼得。每個收集器都是在三角形裡選一個角站。

### Serial 與 Parallel：全停頓的兩代

兩者都是「STW 期間完成全部回收」的古典設計，差別只在回收時用幾條 thread（Serial 一條、Parallel 多條——實測最長停頓 37.5ms vs 16.3ms 就是多核的貢獻）。

別急著把它們掃進歷史：**Parallel 至今是吞吐王**——沒有併發階段的簿記負擔，CPU 全花在刀口上。夜間批次、離線報表這種「沒人在等回應、只求總時間短」的場景，`-XX:+UseParallelGC` 依然是正解。

### G1：用 region 把停頓「配額化」

G1 的三個機關：

1. **Region 化**：heap 切成幾千個等大的 region（1–32MB），Eden/Survivor/Old 不再是連續大區，而是 region 的**邏輯集合**——[分代](gc-basics-and-generations.md)還在，但變成了貼紙
2. **併發標記**：找活人的工作大部分與應用並行（不停頓），只在起訖點短暫 STW
3. **增量回收＋停頓目標**：每次只回收「一部分」region——挑**垃圾比例最高**的下手（Garbage-First 之名由此來），挑多少塊由停頓預算（`-XX:MaxGCPauseMillis`，預設 200ms）反推

實測的性格顯現：G1 停頓**次數最多**（62 次）但**單次最短**（9.2ms）——它就是刻意把「少次大停頓」換成「多次小停頓」，服務的長尾延遲因此平滑。這是 JDK 9 起的預設，通用場景的均衡選擇。

### ZGC：連「搬家」都不停

G1 還有一個藏不住的停頓來源：**搬移（evacuation）時應用必須停**——物件搬走了，舊參考怎麼辦？ZGC 的回答是兩個機關：

1. **著色指標（colored pointers）**：64-bit 指標挪幾個 bit 出來記「這個參考的 GC 狀態」——中繼資料直接住在指標裡
2. **讀屏障（load barrier）**：應用每次讀參考時做一個極便宜的檢查——發現讀到「搬家中」的物件，**當場自我修正**（幫忙搬完或改指新址）

於是標記、搬移**全部**與應用並行，STW 只剩幾個「換階段」的同步點——**停頓與 heap 大小、存活集大小完全脫鉤**（實測 5 微秒；官方目標 sub-millisecond，TB 級 heap 也一樣）。代價：讀屏障是每次參考讀取的「稅」（吞吐通常降個位數百分比）、footprint 較高。JDK 17 的 ZGC 不分代；**JDK 21 起 Generational ZGC**（JDK 23 起成預設）補回分代紅利，把吞吐差距進一步縮小。

### 選型速查

| 場景 | 選擇 | 理由 |
|---|---|---|
| 一般 Web 服務 | **G1（預設）** | 均衡；200ms 停頓目標對多數 SLA 夠用 |
| 批次／離線運算 | Parallel | 吞吐王；沒人在等回應 |
| 大 heap（數十 GB＋）或嚴格延遲（P99 個位數 ms） | ZGC | 停頓與 heap 脫鉤 |
| 容器記憶體極小（<1GB） | Serial | footprint 最低，簿記最少 |

選完**用 GC log 驗證**（[基礎篇](gc-basics-and-generations.md)教的讀法）——調參的第一步永遠是先量。

## 實際案例

▶ 可執行範例：[GcAlgoBench.java](examples/GcAlgoBench.java)

負載設計：維持約 190MB 的存活輪替窗口、總共攪動 6GB 配置——有垃圾可收、有活人要搬，收集器的性格才顯得出來。同一程式換四個 `-XX:+Use*GC` 旗標實測（Temurin 17、1GB heap）：

```
Serial   ：總耗時 368 ms｜停頓 28 次，最長 37.5 ms
Parallel ：總耗時 232 ms｜停頓 24 次，最長 16.3 ms
G1       ：總耗時 325 ms｜停頓 62 次，最長  9.2 ms
ZGC      ：總耗時 156 ms｜停頓 21 次，最長  0.005 ms
```

三個解讀：

1. **Parallel vs Serial**：同樣的全停頓設計，多執行緒把最長停頓砍半——這是「用人力縮停頓」的極限
2. **G1 的簽名**：停頓次數翻倍、單次縮到 9.2ms——「配額化」的具象。注意它總耗時比 Parallel 長：併發標記和 region 簿記是有價的，這就是三角習題
3. **ZGC 的 5 微秒**：不是量錯——它的 STW 只剩換階段的同步點，回收全程與應用並行。一個誠實的註腳：此規模下 ZGC 總耗時最短**不代表**它吞吐最強（併發工作疊在多核空檔上剛好賺到）；長時間高負載下 Parallel 的吞吐優勢仍在，**看停頓欄位才是這張表的正確讀法**

## 技術優缺點

### 演化的主線：把 STW 逐步趕出關鍵路徑

- Serial → Parallel：**人多**（多執行緒收）
- Parallel → G1：**分批**（region 配額＋併發標記）
- G1 → ZGC：**連搬家都並行**（著色指標＋讀屏障）——STW 從「工作」退化成「打信號」

### 沒有免費的午餐

- G1 的簿記稅：region 間參考（RSet）、併發標記的 CPU——換來停頓可預算
- ZGC 的屏障稅：每次讀參考都付一點——換來停頓與 heap 脫鉤
- **Parallel 零併發稅**——所以吞吐場景它還是王
- 回到實務：**預設 G1 不動它**；有明確的延遲 SLA 或超大 heap 才動手換 ZGC；批次考慮 Parallel——並且每一次更換都用同一份負載跑 GC log 對比（本篇的實驗方法就是模板）

## 小結

- 評估三角：**吞吐／延遲／footprint** 不可兼得——四個收集器是四種取捨
- **Parallel**：吞吐王，批次首選；**G1**（預設）：region 配額化停頓（實測次數多但單次 9.2ms）；**ZGC**：著色指標＋讀屏障讓搬移也並行，停頓 5 微秒且與 heap 脫鉤
- 演化主線一句話：**把 STW 從「做工作」變成「打信號」**
- JDK 21+ 的 Generational ZGC 補回分代紅利——延遲派的版圖還在擴大
- 選型後永遠用 **GC log 實測驗證**——本篇的四旗標對比就是現成模板

## 常見面試題

1. G1 憑什麼控制停頓時間？（提示：region 化、垃圾優先挑選、MaxGCPauseMillis 反推回收量）
2. ZGC 為什麼停頓能與 heap 大小無關？（提示：著色指標＋讀屏障——搬移併發化，STW 只剩換階段）
3. 什麼場景該用 Parallel 而不是 G1／ZGC？（提示：吞吐 vs 延遲的三角；批次無人等回應）
